# 个人文件仓库 - 核心数据模型设计策略

## 1. 设计理念

本系统采用**以文件为核心，属性为维度**的数据管理策略。所有功能（浏览、分类、搜索、筛选）本质上都是对文件记录不同属性的操作和查询。

---

## 2. 核心数据表结构

### 2.1 文件表 (files)
这是系统的核心表，存储所有文件的元数据和状态信息。

**主要字段：**
- **基础属性**：`title`, `file_path`, `file_name`, `file_ext`, `file_size`, `is_folder`
- **状态属性**：`file_exists` (是否存在), `favorite` (是否收藏), `read_status` (阅读状态)
- **描述属性**：`author` (作者), `year` (出版时间), `tags` (标签)
- **关联属性**：`category_id` (所属分类ID)
- **时间属性**：`created_at`, `updated_at`, `last_checked`

### 2.2 分类表 (categories)
用于定义文件的组织类别，支持**无限层级子分类**，并内置**显式层级字段**加速查询。

**主要字段：**
- `id`: 分类唯一标识
- `name`: 分类名称（全局唯一）
- `parent_id`: 父分类ID（NULL表示顶级分类）
- `sort_order`: 同级分类的排序序号（0 起递增）
- `color`: 显示颜色
- `icon`: 图标

**层级关系：**
- 通过 `parent_id` 外键关联自身，形成树形结构
- `sort_order` 由后端在增删改/拖拽移动时自动维护
- 支持拖拽移动分类（`PUT /api/categories/<id>/move`，支持 before/after/into 三种定位）
- 前端通过递归算法渲染树形展示

---

## 3. "浏览"功能的本质：动态属性筛选

侧边栏的"浏览"部分不是独立的数据存储，而是基于文件属性的**实时查询视图**：

| 浏览选项 | 筛选条件 | 说明 |
|---------|---------|------|
| 全部文件 | 无 | 查询所有文件记录 |
| 现存文件 | `file_exists = 1` | 仅显示路径有效的文件 |
| 已丢失 | `file_exists = 0` | 显示源文件已被删除/移动的记录 |
| 收藏 | `favorite = 1` | 显示用户标记为收藏的文件 |
| 未分类 | `category_id IS NULL` | 显示未分配分类的文件 |

**实现方式：**
前端通过传递不同的查询参数（如 `file_exists`, `favorite`, `uncategorized`）到后端 API，后端根据这些参数构建 SQL WHERE 子句进行筛选。

---

## 4. "分类"功能的本质：文件关联属性

分类是文件的一个**外键关联属性**，而非独立的存储容器。

**工作原理：**
1. 每个文件通过 `category_id` 字段与 `categories` 表建立关联
2. 一个文件只能属于一个分类（单值属性）
3. **删除分类时，会递归删除该分类及其所有子分类下的所有文件**（带确认提示）
4. 分类本身只是一个"标签"或"抽屉"的概念，不存储文件副本
5. 支持无限层级的子分类，通过 `parent_id` 字段实现树形结构

**优势：**
- 灵活性高：可以随时更改文件的分类
- 数据一致性：文件只有一份记录，避免冗余
- 易于扩展：可以轻松添加新的分类维度（如标签系统）

## 5. 数据流转示例

### 5.1 添加文件流程
```
用户选择文件 → 获取文件信息 → 创建 files 记录 → 设置 category_id → 存入数据库
```

### 5.2 浏览"收藏"文件流程
```
用户点击"收藏" → 前端请求 /api/files?favorite=1 → 后端执行 SQL: 
SELECT * FROM files WHERE favorite = 1 → 返回结果 → 前端渲染列表
```

### 5.3 按分类查看流程
```
用户点击某分类 → 前端请求 /api/files?category_id=X → 后端执行 SQL: 
SELECT * FROM files WHERE category_id = X → 返回结果 → 前端渲染列表
```

### 5.4 分类层级维护流程
```
分类增删改/拖拽移动 → 后端重新计算同级 sort_order → 批量写入 categories 表
```
该流程完全在 categories 表内完成，**不涉及 files 表的任何操作**。

---

## 6. 关键设计原则

### 6.1 单一数据源
- 所有文件信息只存储在 `files` 表中
- 不存在"收藏表"、"丢失文件表"等冗余存储
- 不同视图只是同一数据的不同呈现方式

### 6.2 属性驱动
- 文件的所有特征都通过属性字段表达
- 新功能可以通过添加新属性或新筛选条件实现
- 例如：未来可以添加 `priority`（优先级）、`project`（所属项目）等属性

### 6.3 相对路径存储
- `file_path` 字段存储相对于程序运行目录的路径
- 便于项目迁移和便携使用
- 通过 `to_relative_path()` 和 `to_absolute_path()` 函数转换

### 6.4 软删除机制
- 从仓库移除文件时，只删除数据库记录
- 不会删除磁盘上的实际文件
- `file_exists` 字段用于标记文件是否仍可访问

### 6.5 级联删除机制
- **删除分类时**：递归获取所有子分类ID，然后删除这些分类下的所有文件
- **防循环引用**：编辑分类时，排除自己和所有子分类作为父选项
- **确认提示**：删除分类前必须通过二次确认弹窗，明确告知将删除所有文件

---

## 7. 扩展性思考

基于当前设计，可以轻松扩展以下功能：

1. **多标签系统**：已有 `tags` 字段，可实现类似分类但更灵活的多标签管理
2. **自定义属性**：新增扩展表中存储用户自定义字段
3. **文件版本管理**：添加 `version`、`parent_id` 等字段追踪文件历史
4. **共享与协作**：添加 `owner_id`、`shared_with` 等字段支持多用户
5. **智能分类**：基于文件名、路径、内容自动推荐分类
6. **文件夹归档**：`is_folder` 字段区分文件/文件夹，文件夹作为快捷目录链接管理

---

## 8. 技术实现要点

### 8.1 数据库优化
- 使用 SQLite WAL 模式提高并发性能
- 为常用查询字段建立索引（`category_id`, `title`, `tags`, `file_exists`, `parent_id`）
- 启用外键约束保证数据完整性

### 8.2 API 设计
- RESTful 风格：`GET /api/files`, `POST /api/files`, `PUT /api/files/{id}`
- 统一响应格式：`{ data, message, error }`
- 支持分页、排序、多条件筛选
- 分类排序接口：`PUT /api/categories/<id>/move`，传入目标分类和位置（before/after/into）

### 8.3 前端交互
- 状态管理：`currentFilter`, `currentCategory`, `currentPage`
- 防抖搜索：输入后 300ms 才触发查询
- 局部更新：收藏、阅读状态等操作无需刷新整个列表
- 自定义下拉组件：作者（多值分号分隔）、阅读状态均使用统一风格的自定义下拉，互斥展示
- 列布局：收藏 ⭐ → 出版时间 → 标题（弹性列）→ 格式 → 作者 → 阅读状态
- 拖拽排序：侧边栏分类支持拖拽移动和同级排序，后端自动维护 `sort_order`

---

## 9. 总结

本系统的核心思想是：**文件是实体，属性是维度，浏览是视角，分类是标签**。

`is_folder` 字段将文件夹视为一种特殊类型的"文件"，完全复用现有分类、搜索、收藏等所有功能，仅在前端通过图标区分展示。

这种设计使得系统具有高度的灵活性和可扩展性，所有功能都围绕文件的属性展开，避免了数据冗余和复杂的同步逻辑。
